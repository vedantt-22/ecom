Step 1: Project Scaffolding & Backend Structure
The Objective
    We're initialising the Node.js/Express backend with a clean, professional folder structure and the correct dependencies. 
    This isn't just "running npm init" — the architectural decisions you make here (how you organise folders, how you separate 
    concerns) directly signal your seniority to a reviewer. Every folder you create has a reason that you must be able to defend.

Step 2: Database Schema Design & TypeORM Entities
The Objective
    We're designing the entire relational schema and translating it into TypeORM entity classes. This is the most consequential 
    architectural decision in the project — a flawed schema cannot be patched later without painful migrations. 
    Every column, every foreign key, and every constraint needs a reason you can articulate.

Step 3: TypeORM DataSource Configuration & Database Connection
The Objective
    We're wiring TypeORM to the actual database and creating the single DataSource instance that the entire backend will use. 
    This step also establishes how we handle environment variables safely, and makes a critical architectural decision: 
    synchronize vs migrations. You must be able to defend this choice confidently.

Step 4: Seed Data — Taxonomy Tree & Admin Account
The Objective
    We're writing a script that populates the database with an initial taxonomy (Types → Categories → SubCategories), sample 
    products, and a seeded Admin account. This serves two purposes: it gives you working data to test every feature against, 
    and it demonstrates that you understand idempotent scripts — a script safe to run multiple times without duplicating data.

Step 5: Express App Structure — Middleware Stack & Routing Skeleton
The Objective
    We're building the complete Express application skeleton — every route barrel file, the middleware execution order, 
    and the global error handler. After this step, your server has a fully wired request pipeline even though most route 
    handlers will return placeholder responses. This matters because it forces you to think about the shape of your API 
    before writing any business logic.

Step 6: In-Memory Session Store & Session Middleware
The Objective
    We're building the custom session store — a plain Node.js Map object that tracks every active login. 
    This is the security mechanism that makes immediate account lock enforcement possible. 
    It is also the most architecturally interesting piece of the auth system and the one your manager will probe hardest, 
    because it's a deliberate, justified choice that deviates from the standard express-session library approach.

Step 7: Registration Endpoint & bcrypt Password Hashing
The Objective
    We're implementing the first real business logic — customer registration. This step covers input validation, 
    duplicate email detection, secure password hashing, and a clean JSON response. 
    By the end, POST /api/auth/register will fully work and you'll be able to create customer accounts that persist 
    in the database.

Step 8: Login, Token Issuance & HTTP-only Cookie
The Objective
    We're implementing login — the most security-critical endpoint in the entire application. 
    A successful login does three things atomically: verifies the password with bcrypt, issues a signed JWT, and 
    registers the session in the Map store. All three must succeed together or the login fails cleanly. 
    By the end of this step, POST /api/auth/login will work, set an HTTP-only cookie, and GET /api/auth/me will 
    return the authenticated user's profile.

Step 9: RBAC Middleware — already built in Step 6

Step 10: Product APIs — Taxonomy, Listing, Search & Filtering
The Objective
    We're implementing the three product endpoints that power the entire shopping experience. 
    This is where 8 marks for search and 8 marks for filtering/pagination are earned. 
    The spec has a specific, testable requirement: searching "table" must return both "Wooden Table" (Furniture) and
    "Multiplication Table Book" (Stationery) — cross-taxonomy results in one response.

Step 11: Image Upload with Multer & express.static Serving
The Objective
    We're implementing product image uploads and static file serving. The spec is very explicit about this: images live in a 
    ProductImages folder on disk, the database stores only the relative file path, and express.static serves them directly — no 
    controller involved in serving the image itself. This is worth 5 marks and has three distinct testable requirements: upload works, 
    default fallback works, and the path stored in the DB resolves correctly via the static URL.

Step 12: Cart API — Persistent Cart with Quantity Management
The Objective
    We're implementing the persistent cart — the feature that earns 8 marks and has three specific, testable 
    requirements from the spec: items persist across logout/login, quantity management works correctly, and the cart 
    clears completely after a successful order. The persistence requirement is what makes this architecturally 
    interesting — the cart lives in the database tied to the user's ID, not in a browser session or localStorage.

Step 13: Checkout & Order Creation — The Price Snapshot Pattern
The Objective
    We're implementing checkout — the most data-integrity-critical endpoint in the entire application. 
    This is where the price snapshot pattern from Step 2 pays off. A successful checkout must do five things
    atomically: validate stock for every item, create the Order record, create individual OrderItem records 
    with prices frozen at the current moment, decrement stock, and clear the cart. If any step fails, none 
    of them should commit. This is where database transactions become mandatory.
