---Auth.Middleware---

1. requireAuth Function
This function checks if the user is Authenticated (Who are you? Do you have a valid ID card/JWT?).

Line-by-Line Breakdown:
passport.authenticate("jwt", { session: false }, ...): We tell Passport to use the JWT strategy. We set session: false because we aren't using old-school cookies/sessions; we are using modern Tokens.

The Callback (err, user, info):

err: If the database crashes or something explodes, we return a 500 Internal Server Error.

!user: If the token is expired, fake, or missing, we return a 401 Unauthorized. We send a JSON message so your Angular frontend knows exactly why the login failed.

req.user = user: This is a crucial step. If the user is valid, we "attach" them to the request object. Now, every following function knows exactly which user is making the request.

next(): This tells Express: "Everything looks good, move to the next function in line."

)(req, res, next): This looks weird, right? passport.authenticate actually creates a function. This final part immediately executes that function with the current request details.

2. requireRole Function
This is a Middleware Factory. It doesn't just check if you are logged in; it checks if you are Authorized (Are you an Admin? A Customer?).

export function requireRole(...allowedRoles: string[]): The ... is a "rest parameter." It allows you to pass one or many roles (e.g., requireRole("admin", "manager")).
return (req, res, next) => { ... }: This function creates another function. We do this so we can "save" the allowedRoles inside the middleware.
const user = req.user as any: It grabs the user we attached earlier in requireAuth.
!allowedRoles.includes(user.role): It checks: "Is the user's role in the list of roles allowed for this specific route?"
res.status(403): If a regular customer tries to enter the /admin panel, they get a 403 Forbidden.
Note: 401 means "I don't know who you are." 403 means "I know who you are, but you aren't allowed in here."


Here is a breakdown of the logic, the dependencies, and the data flow.
1. Dependency Injection via inject() In functional guards, you cannot use a constructor. 
Instead, you use the inject() function to grab your services from the Angular dependency injection container.
AuthService: Used to check the current user's login status.Router: Used to redirect the user to a different page (the login page) if they aren't authorized.
2. The Logic Flow - The guard doesn't just return a simple true or false; it returns an Observable. Angular will wait for this Observable to emit a value before finishing the navigation.
Step-by-Step Execution - Subscription: The router "subscribes" to the currentUser$ stream from your AuthService.take(1): This is crucial. It tells the stream to take the first available value and then immediately complete. If you don't use take(1), the guard remains "open," and the navigation will hang forever because it's waiting for the stream to finish.map(): This transforms the "User" object into a result the Router understands:If user exists: It returns true. The guard "resolves," and the user sees the page.If user is null/undefined: It returns a UrlTree (via router.createUrlTree(['/login'])).Note on UrlTree: In older Angular, we used to manually call router.navigate(). In modern Angular, returning a UrlTree is the preferred way. It tells the router: "Stop going where you were going, and go here instead."
3. The Sequence of Events - When a user clicks a link to a protected route (e.g., /dashboard), the following happens:PhaseActionTriggerUser navigates to a route protected by authGuard.CheckThe guard function runs and asks AuthService for the currentUser$.ProcessingThe RxJS operators (take and map) process the user data.Decision AUser is logged in $\rightarrow$ Returns true $\rightarrow$ Component loads.Decision B User is guest $\rightarrow$ Returns /login UrlTree $\rightarrow$ Router cancels original request and redirects.
4. Why use Observable<boolean | UrlTree>?By using an Observable, your guard can handle asynchronous checks. For example, if your AuthService needs to ping a server to verify a session token, the UI won't "flicker" or load the secret page halfway; it waits until the server responds before deciding to show the page or boot the user to login.Summary of BenefitsPerformance: No class instance needs to be created.Readability: The logic is contained in one clear function.Type Safety: CanActivateFn ensures you provide the correct parameters (route and state) even if you don't use them in the code.
